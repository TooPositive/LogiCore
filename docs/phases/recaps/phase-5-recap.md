# Phase 5 Technical Recap: Assessment Rigor — Judge Bias, Drift Detection & Prompt Caching

## What This Phase Does (Business Context)

Phase 4's quality dashboard says "Context Precision: 0.92." But the judge scoring that metric (GPT-5-mini) is evaluating output generated by GPT-5.2 — same OpenAI family. Published benchmarks show 10-15% self-preference inflation. Meanwhile, Azure silently updates model versions and nobody notices until retrieval quality drops. Phase 5 builds the evaluation pipeline's immune system: three independent bias detectors, automated drift detection with graduated alerting, and honest prompt caching metrics that account for RBAC partition fragmentation.

## Architecture Overview

```
Quality Pipeline (Pillar 1: Judge Bias Mitigation)
  │
  ├── JudgeConfig: model family separation enforcement
  │     └── ModelFamily enum → get_model_family() → validate_judge_generator_independence()
  │     └── register_model_family() for fine-tuned/custom model names
  │
  ├── PairwiseScorer: position-swap agreement
  │     round1: judge(query, A, B) → raw verdict
  │     round2: judge(query, B, A) → translate back to answer labels
  │     agree? → valid result / disagree? → position bias → discard
  │
  ├── BiasDetector: three detection methods
  │     ├── detect_position_bias(): PairwiseScorer on n pairs → disagreement rate
  │     ├── detect_verbosity_bias(): short correct vs long wrong → length preference rate
  │     └── detect_self_preference(): same-family vs cross-family → scaled 0.0-1.0
  │
  ├── HumanCalibration: golden set validation
  │     └── 50 human-scored entries → Spearman rank correlation → PASS (≥0.85) / HALT
  │
  └── BootstrapCI: confidence intervals on eval scores
        └── Non-parametric resampling → 95% CI → "0.83 ± 0.09 might be below gate"

Drift Detection (Pillar 2)
  │
  ├── ModelVersionRegistry: in-memory version tracking + baseline scores
  │     └── register() → get_current_version() → get_baseline_scores() → detect_version_change()
  │
  └── DriftDetector: regression detection + alerting
        ├── check_regression(): per-metric delta → classify_drift_severity()
        │     GREEN (<2%) → log only
        │     YELLOW (2-5%) → investigate within 24h
        │     RED (>5%) → halt quality gates
        └── AlertHandler ABC → LogAlertHandler (default), Slack/PagerDuty (implement protocol)

Prompt Caching (Pillar 3)
  │
  ├── PromptOptimizer: static-first restructuring
  │     └── restructure() → sort by STATIC → SESSION_STABLE → DYNAMIC

  │
  └── CacheMetrics: RBAC-aware partition tracking
        └── record_query(partition_key) → first query = miss, subsequent = hit

```

## Components Built

### 1. Judge Configuration: `apps/api/src/telemetry/judge_config.py`

**What it does**: Identifies model families from model name strings and enforces cross-family independence between judge and generator models.

**The pattern**: Strategy Pattern via configurable lookup + Registry Pattern for overrides. The family detection uses two resolution layers: (1) exact-match overrides from a runtime registry, (2) prefix matching against a static pattern dict. This two-tier approach handles both standard models (prefix) and deployment-specific names (exact match).

**Key code walkthrough**:
```python
# apps/api/src/telemetry/judge_config.py:67-98
def get_model_family(
    model_name: str,
    family_patterns: dict[str, ModelFamily] | None = None,
) -> ModelFamily:
    lower_name = model_name.lower()

    # 1. Check exact-match overrides first (for fine-tuned/custom names)
    if lower_name in _EXACT_FAMILY_OVERRIDES:
        return _EXACT_FAMILY_OVERRIDES[lower_name]

    # 2. Prefix matching against known patterns
    patterns = family_patterns or FAMILY_PATTERNS
    for prefix, family in patterns.items():
        if lower_name.startswith(prefix):
            return family

    # 3. Unknown = fail closed on independence check
    return ModelFamily.UNKNOWN
```

**Why it matters**: Without family-level separation, GPT-5-mini judging GPT-5.2 looks like "different model" but is same-family self-preference. The function costs EUR 0.00 (string comparison) but changes the fundamental reliability of every quality metric in the system.

**Alternatives considered**:
- **Model-level only** (just check model names differ): Misses same-family bias. Published benchmarks show 10-15% inflation within a family.
- **API-based family detection** (call model metadata endpoint): Adds latency and external dependency for a string comparison. Rejected.
- **Hardcoded allowlist** (explicit judge-generator pairs): Brittle, doesn't scale to new models. Prefix matching is more resilient.

**The independence check** (`validate_judge_generator_independence`, lines 101-130):
- UNKNOWN family always fails → safe default. Blocks judging rather than allowing potentially biased judging.
- Both models must be known AND different families to pass.
- This is why `register_model_family()` exists: deployment-specific names that don't match prefixes need explicit registration.

### 2. Pairwise Scorer: `apps/api/src/telemetry/quality_pipeline.py` (lines 83-146)

**What it does**: Runs each A/B comparison twice with swapped positions. Translates the second round's positional verdict back to answer labels. Requires agreement for a valid result.

**The pattern**: Double-checking with Translation. Not a standard GOF pattern but a domain-specific technique for detecting position bias. The judge function is injectable (Callable[[str, str, str], str]) — any function that takes (query, first, second) and returns "A"/"B"/"TIE" works. This makes the scorer testable with deterministic mock judges.

**Key code walkthrough**:
```python
# apps/api/src/telemetry/quality_pipeline.py:109-130
# Round 1: A is first, B is second
round1_raw = self._judge_fn(query, answer_a, answer_b)

# Round 2: B is first, A is second
round2_raw = self._judge_fn(query, answer_b, answer_a)

# Translate round 2 result back to answer labels
# In round 2: "A" means first position = answer_b
if round2_raw == "A":
    round2_translated = "B"  # Judge picked position 1, which was answer B
elif round2_raw == "B":
    round2_translated = "A"  # Judge picked position 2, which was answer A
else:
    round2_translated = "TIE"
```

**Why the translation matters**: The judge always reports in positional terms ("A" = first shown, "B" = second shown). In round 2, the positions are swapped. So if the judge says "A" in round 2, it picked answer_b (which is now in position 1). The translation converts positional results back to answer labels for comparison.

**The consistency check** (ComparisonResult, lines 53-80):
- `round1_winner == round2_winner` → both rounds picked the same ANSWER → consistent
- `round1_winner != round2_winner` → one or both rounds picked by POSITION → bias detected
- Exception: TIE/TIE is consistent (both rounds agreed it's a tie)

**Why discard instead of average**: Averaging a biased result with its mirror produces a number that looks reasonable but hides the fact that the judge wasn't evaluating quality at all. Discarding maintains signal integrity: every result in the dataset was position-independent. The cost is lower sample size, which Bootstrap CI quantifies.

### 3. Bias Detector: `apps/api/src/telemetry/quality_pipeline.py` (lines 149-312)

**What it does**: Three independent detection methods for position, verbosity, and self-preference biases. Each returns a rate (0.0-1.0). `full_bias_report()` combines all three with pre-computed Spearman correlation into a `JudgeBiasResult`.

**The pattern**: Dependency Injection for the judge function. All three detectors share the same `_judge_fn: JudgeFn` which is passed at construction. This is the key to testability — mock judges with known biases validate the detection logic without needing real LLM calls.

**Verbosity bias walkthrough** (lines 186-233):
```python
# Convention: answer_a = short correct, answer_b = long wrong
for query, short_answer, long_answer in pairs:
    short_len = len(short_answer)
    long_len = len(long_answer)

    # Skip pairs without meaningful length difference
    ratio = max(short_len, long_len) / min(short_len, long_len)
    if ratio < self._verbosity_length_ratio:  # configurable, default 1.5x
        continue

    eligible_count += 1
    result = self._judge_fn(query, short_answer, long_answer)
    if result == "B":  # Picked longer answer (wrong) -> verbosity bias
        biased_count += 1
```

**Why the threshold is configurable**: Legal domains need thorough answers — a 1.2x ratio might be appropriate. Operational dashboards need concise answers — 2.0x avoids flagging legitimately detailed responses. The default 1.5x was chosen as a middle ground.

**Self-preference scaling** (lines 270-280): A fair judge picks same-family 50% of the time. Raw rate of 0.5 = no bias. Scaling `(raw - 0.5) / 0.5` maps 0.5 → 0.0 and 1.0 → 1.0, making the metric directly interpretable. Rates ≤ 0.5 (inverse preference) are reported as 0.0.

### 4. Human Calibration: `apps/api/src/telemetry/quality_pipeline.py` (lines 315-376)

**What it does**: Computes Spearman rank correlation between human expert scores and LLM judge scores. Returns PASS/HALT based on configurable minimum correlation (default 0.85).

**The pattern**: Quality Gate with Binary Output. Not probabilistic — either the judge is calibrated (correlation ≥ threshold) or it isn't (HALT everything). This is deliberately conservative: better to stop automated quality gates than to operate with an unreliable judge.

**The Spearman implementation** (`_spearman_rank_correlation`, lines 378-406):
```python
# Uses Pearson correlation of ranks — handles ties correctly
ranks_x = _compute_ranks(x)
ranks_y = _compute_ranks(y)
mean_x = sum(ranks_x) / n
mean_y = sum(ranks_y) / n

cov = sum((rx - mean_x) * (ry - mean_y) for rx, ry in zip(ranks_x, ranks_y))
var_x = sum((rx - mean_x) ** 2 for rx in ranks_x)
var_y = sum((ry - mean_y) ** 2 for ry in ranks_y)

# Zero variance = all tied = correlation undefined -> 0.0 (conservative)
if var_x == 0 or var_y == 0:
    return 0.0
return cov / (var_x**0.5 * var_y**0.5)
```

**Why Pearson-of-ranks instead of the d² formula**: The classic Spearman formula `rho = 1 - (6*Σd²)/(n*(n²-1))` is an approximation that breaks on tied values. When all human scores are identical (zero variance), the d² formula returns a non-zero value (wrong). Pearson-of-ranks returns 0.0 explicitly because correlation is mathematically undefined with zero variance.

**Why not scipy.stats.spearmanr**: No external dependency for a 30-line function. The implementation is a standard algorithm, easily replaced with `scipy.stats.spearmanr` as a one-line swap if scipy is already a dependency.

### 5. Bootstrap Confidence Intervals: `apps/api/src/telemetry/quality_pipeline.py` (lines 430-487)

**What it does**: Computes non-parametric confidence intervals on evaluation scores via bootstrap resampling.

**The pattern**: Monte Carlo Estimation. Resample with replacement N times, compute mean of each resample, take percentiles. No assumptions about score distribution (important because eval scores are often skewed — many high scores, few low).

**Why bootstrap instead of Gaussian CI**: Standard CI assumes normality. Eval score distributions are often bimodal (a model either gets a question right or completely wrong). Bootstrap works on any distribution. The cost: N=1000 resamples × len(scores) operations, which is microseconds for typical eval datasets.

### 6. Model Version Registry: `apps/api/src/telemetry/model_registry.py`

**What it does**: In-memory model version tracking with baseline score CRUD and version change detection. Each model has a current version and a full history.

**The pattern**: Repository Pattern (in-memory). The interface (register, get_current, get_baseline, detect_change, get_history) is identical to what a PostgreSQL-backed implementation would expose. For production, implement the same 4 methods against a database table.

**Key design choices**:
- `detect_version_change()` returns True for unregistered models (first time = change, which is safe)
- `update_baseline()` modifies in-place instead of creating new history entry — baselines can be refined without inflating version history
- `list_models()` provides registry enumeration for multi-model deployments

### 7. Drift Detector: `apps/api/src/telemetry/drift_detector.py`

**What it does**: Compares current quality scores against registered baselines, classifies deltas into green/yellow/red severity, and dispatches alerts via an extensible handler protocol.

**The pattern**: Observer Pattern via AlertHandler ABC. The detector generates alerts; handlers consume them. Adding Slack notifications means implementing one method (`handle(alert: DriftAlert)`), not modifying detection logic.

**Key code walkthrough**:
```python
# apps/api/src/telemetry/drift_detector.py:26-52
def classify_drift_severity(
    delta: float,
    yellow_threshold: float = 0.02,
    red_threshold: float = 0.05,
) -> DriftSeverity:
    if delta >= 0:
        return DriftSeverity.GREEN  # Improvements never trigger alerts

    abs_delta = abs(delta)
    if abs_delta >= red_threshold:
        return DriftSeverity.RED
    if abs_delta >= yellow_threshold:
        return DriftSeverity.YELLOW
    return DriftSeverity.GREEN
```

**Critical design: per-metric, not aggregate**. The `check_regression()` method (lines 118-171) iterates each metric individually. A 7% precision drop and 2% relevancy improvement look like -2.5% aggregate. Aggregate masking is the enemy of drift detection. The test suite includes a specific test (`test_metrics_cancelling_out_still_individual_alerts`) that proves 5 regressions + 5 improvements still generate 5 alerts.

### 8. Prompt Optimizer: `apps/api/src/telemetry/prompt_optimizer.py`

**What it does**: Restructures prompt templates for cache-optimal ordering (static first, dynamic last), scores cache-friendliness, tracks per-partition cache metrics, and estimates daily cost savings.

**The pattern**: Strategy Pattern for section classification + Metrics Collection for cache tracking. Sections are labeled with a `SectionType` enum (STATIC/SESSION_STABLE/DYNAMIC). The restructure is a stable sort by type order, preserving relative order within each type.

**Cache-friendliness score** (lines 205-252):
```python
# Factor 1: what fraction of tokens are cacheable
cacheable_ratio = cacheable_tokens / total_tokens

# Factor 2: how much cacheable content is in the prefix (before first dynamic)
prefix_cacheable = 0
for section in sections:
    if section.section_type == SectionType.DYNAMIC:
        break
    prefix_cacheable += section.estimated_tokens

prefix_ratio = prefix_cacheable / cacheable_tokens
return cacheable_ratio * prefix_ratio
```

**Why two factors matter**: A prompt with 80% static content gets `cacheable_ratio=0.8`. But if half the static content is AFTER the dynamic content, only 40% is in the prefix. The cache key is a prefix hash — anything after the first dynamic token is recomputed. The score is `0.8 × 0.5 = 0.4`, which correctly reflects that only 40% is actually cached.

**CacheMetrics and RBAC honesty** (lines 96-171): Each RBAC partition (clearance + department + entity) creates a separate cache prefix. CacheMetrics tracks `unique_partitions` as the real fragmentation metric. The spec claimed 60% hit rate; multi-tenant reality is 15-25% because 5 tenants = 5 cold misses per burst, not 1.

### 9. Domain Models: `apps/api/src/domain/telemetry.py` (lines 151-258)

**What it does**: Five Pydantic v2 models for Phase 5: JudgeBiasResult, DriftSeverity, DriftAlert, ModelVersion, PromptCacheStats.

**The pattern**: Value Objects with computed properties. JudgeBiasResult.is_calibrated is a computed AND across four conditions (position, verbosity, self-preference rates below thresholds + Spearman above minimum). All thresholds are configurable per deployment.

### 10. CLI Scripts: `scripts/calibrate_judge.py` + `scripts/run_drift_check.py`

**What they do**: cron-compatible CLI tools with semantic exit codes for CI/CD integration.

**The pattern**: Exit Code Protocol. calibrate_judge.py: 0=PASS, 1=HALT, 2=ERROR. run_drift_check.py: 0=GREEN, 1=RED, 2=YELLOW, 3=ERROR. These compose with shell scripting (`if python scripts/run_drift_check.py; then ...`) and CI systems (non-zero exit = pipeline failure).

## Key Decisions Explained

### Decision 1: Family-Level Separation (Not Model-Level)

- **The choice**: Judge and generator must be from different model FAMILIES (OpenAI vs Anthropic), not just different models (GPT-5-mini vs GPT-5.2)
- **The alternatives**: (a) Model-level check only (different model name), (b) No check (trust the user), (c) Embedding similarity check between model outputs
- **The reasoning**: Published benchmarks measure 10-15% self-preference within model families. GPT-5-mini and GPT-5.2 share training data, RLHF preferences, and output style. A model-level check would approve this pairing.
- **The trade-off**: Claude Sonnet as judge costs the same (EUR 0.011/eval) so there's no cost trade-off. The only trade-off is complexity in the family detection code.
- **When to revisit**: If a provider demonstrates no self-preference within their family via independent benchmarks, model-level could suffice.
- **Interview version**: "We enforce family-level separation between judge and generator — GPT-5-mini judging GPT-5.2 is still same-family self-preference. Published benchmarks show 10-15% score inflation. The fix is a string prefix comparison that costs zero but changes the reliability of every quality metric. Claude Sonnet as judge is the same price."

### Decision 2: Discard Disagreements, Don't Average

- **The choice**: When pairwise rounds disagree, throw out the result instead of averaging
- **The alternatives**: (a) Average the two rounds, (b) Take the majority of 3+ rounds, (c) Flag but include in dataset
- **The reasoning**: Averaging a position-biased result with its mirror produces a number that looks neutral but hides the fact that the judge was evaluating position, not quality. Discarding preserves signal integrity — every remaining result was verified position-independent.
- **The trade-off**: Lower sample size. If 20% of comparisons are position-biased, you lose 20% of your data.
- **When to revisit**: If position bias rate is very low (<5%), averaging might be acceptable since the masked bias is negligible.
- **Interview version**: "Disagreement between swapped rounds means the judge picked by position, not quality. We discard those results rather than averaging because averaging masks the bias — the aggregate number looks fine but individual results are unreliable. Bootstrap CI on the remaining results tells you if the reduced sample size is still large enough to trust."

### Decision 3: Three-Tier Drift Severity

- **The choice**: GREEN (<2%), YELLOW (2-5%), RED (>5%) with per-metric alerting
- **The alternatives**: (a) Single 5% threshold, (b) Percentage-based tiers (e.g., relative to baseline), (c) Statistical significance testing per metric
- **The reasoning**: Single threshold has two failure modes: alert fatigue (treating 2.5% as 5%-level) and missed moderate regressions that compound. Three tiers give ops teams a graduated playbook: ignore/investigate/halt.
- **The trade-off**: More configuration parameters (yellow and red thresholds per deployment). Slightly more complex alerting logic.
- **When to revisit**: If ops teams ignore YELLOW alerts consistently, collapse to two tiers.
- **Interview version**: "We use three-tier severity for drift: green under 2% is normal variance, yellow 2-5% gets investigated within 24 hours, red above 5% halts quality gates. Single-threshold systems cause alert fatigue — ops teams start ignoring everything. Per-metric alerting prevents aggregation masking where a 7% precision drop is hidden by a 2% relevancy improvement."

### Decision 4: RBAC Cache Honesty (15-25%, Not 60%)

- **The choice**: Track `unique_partitions` as the fragmentation metric, honestly report multi-tenant hit rate
- **The alternatives**: (a) Report single-tenant 60% and footnote the limitation, (b) Ignore RBAC interaction entirely, (c) Build cross-partition cache sharing with security review
- **The reasoning**: RBAC partitioning (clearance + department + entity) creates separate cache prefixes per tenant. 5 tenants = 5 cold misses per burst, not 1. Reporting 60% for a multi-tenant system is misleading.
- **The trade-off**: Lower reported numbers. The optimization is still free (just reorder the prompt template).
- **When to revisit**: If a single-tenant deployment is the target, 55-65% is achievable.
- **Interview version**: "The spec said 60% cache hit rate. I looked at what RBAC does to cache prefixes — each clearance-department-entity combination creates a separate partition. Multi-tenant hit rate is 15-25%, not 60%. The optimization is still free — you're just reordering prompt sections — but the savings projections need honest partition accounting."

### Decision 5: Spearman via Pearson-of-Ranks

- **The choice**: Implement Spearman as Pearson correlation of ranks rather than the d² approximation
- **The alternatives**: (a) Classic d² formula: `rho = 1 - 6*Σd²/(n*(n²-1))`, (b) scipy.stats.spearmanr, (c) Kendall's tau
- **The reasoning**: The d² formula breaks on heavily tied values — when all human scores are identical (zero variance), it returns a non-zero result. Pearson-of-ranks handles this correctly by checking for zero variance and returning 0.0.
- **The trade-off**: Slightly more code than the d² formula (need `_compute_ranks` helper with average-rank tie handling).
- **When to revisit**: If scipy becomes a dependency for other reasons, replace with `scipy.stats.spearmanr` (identical semantics).
- **Interview version**: "We compute Spearman as Pearson correlation of ranks rather than the classic d-squared formula because d-squared breaks on tied values. When all human scores are identical — which happens when a model is uniformly good or bad — d-squared returns non-zero when correlation is undefined. Pearson-of-ranks returns 0.0, which is the conservative correct answer."

## Patterns & Principles Used

### 1. Dependency Injection via Callable

**What**: Pass a function as a constructor argument instead of hardcoding a dependency.
**Where**: `PairwiseScorer.__init__(judge_fn: JudgeFn)` and `BiasDetector.__init__(judge_fn: JudgeFn)` in `quality_pipeline.py`.
**Why**: The bias detection framework must work with any judge — real LLM calls in production, deterministic mock functions in tests. Injecting the judge function makes the entire pipeline testable without credentials or network calls.
**When not to use**: When the dependency is always the same (e.g., math operations that don't vary by environment).

### 2. Abstract Base Class (ABC) Protocol

**What**: Define an interface that concrete classes must implement.
**Where**: `AlertHandler` ABC in `drift_detector.py:55-68`, implemented by `LogAlertHandler`.
**Why**: Drift alerts need to reach different destinations (logs, Slack, PagerDuty). The ABC ensures any handler has the `handle(alert)` method. Adding Slack: implement one method, register the handler, detection logic unchanged.
**When not to use**: When there's only one implementation and no reasonable future alternatives.

### 3. Registry Pattern (with Override Layer)

**What**: A lookup table with a secondary override layer for special cases.
**Where**: `FAMILY_PATTERNS` (prefix dict) + `_EXACT_FAMILY_OVERRIDES` (runtime dict) in `judge_config.py`.
**Why**: Standard models follow prefix patterns (`gpt-` → OPENAI). Fine-tuned models (`ft:gpt-5.2:logicore:2026`) and Azure deployment names don't. The override layer handles exceptions without complicating the default path.
**When not to use**: When all values follow a single pattern. The override layer adds complexity.

### 4. Fail-Closed Defaults

**What**: Unknown states default to the most restrictive behavior.
**Where**: `ModelFamily.UNKNOWN` fails the independence check in `validate_judge_generator_independence()`.
**Why**: An unrecognized model name should block judging (safe) rather than allow it (potentially biased). This is the security principle applied to evaluation quality: when in doubt, don't trust the result.
**When not to use**: When false negatives are expensive and false positives are cheap. Here, false negative (blocking valid judging) is cheap (just register the model), false positive (allowing biased judging) corrupts all quality metrics.

### 5. Stable Sort for Order Preservation

**What**: Python's `sorted()` with a key function preserves relative order of equal elements.
**Where**: `PromptOptimizer.restructure()` in `prompt_optimizer.py:184-203`.
**Why**: Multiple STATIC sections should maintain their original relative order (system prompt before tool definitions before few-shot examples). The sort key only considers section type, so equal-type sections keep their input order.
**When not to use**: When the full order is deterministic from the key (no equal elements).

### 6. Computed Properties on Pydantic Models

**What**: `@property` on a Pydantic BaseModel for derived values.
**Where**: `JudgeBiasResult.is_calibrated`, `ComparisonResult.is_consistent`, `ComparisonResult.winner`, `PromptAnalysis.static_token_ratio`.
**Why**: Derived values that depend on multiple fields should be computed, not stored. `is_calibrated` is an AND of four conditions — storing it separately risks staleness if any bias rate is updated.
**When not to use**: When the computation is expensive and caching is needed.

## Benchmark Results & What They Mean

### Position Bias Detection

- **What was tested**: 5 biased mock judges (always picks position 1) + 5 fair mock judges (picks better answer) + 2 mixed-signal scenarios
- **Key numbers**: 100% detection on biased, 0% false positives on fair, mixed-signal judges correctly identified
- **Architecture decision**: Pairwise-with-swap is the mandatory approach for CI gates. Single-pass is acceptable for monitoring dashboards where volume averages out bias.
- **Boundary**: All testing uses mock judges. Real LLM position bias rates (published: 10-30%) are validated in Phase 12.

### Self-Preference Scaling

- **What was tested**: Fair judges (50/50 picks), fully biased (always picks same-family), partial bias (70/30), ties excluded
- **Key numbers**: Scaled rate correctly maps 0.5 raw → 0.0 (fair), 1.0 raw → 1.0 (fully biased), 0.7 raw → 0.4 (40% self-preference)
- **Architecture decision**: The EUR 0.00 family-level check is mandatory for any deployment. No cost trade-off.

### Drift Severity Boundaries

- **What was tested**: 10 boundary cases including exact thresholds (-0.02, -0.049, -0.05), improvements, and custom thresholds + 5 large-scale 100-metric scenarios
- **Key numbers**: Exact -2% = YELLOW, exact -5% = RED, any improvement = GREEN. 100 metrics with mixed drift generates exactly 15 alerts (5 RED + 10 YELLOW). Cancelling-out metrics still fire individual alerts.
- **Architecture decision**: Per-metric alerting is non-negotiable. Aggregate-based detection masks regressions.

### RBAC Cache Fragmentation

- **What was tested**: Single partition (99% hit after cold miss), 5 partitions (5 cold misses / 100 queries), 20+ partitions
- **Key numbers**: Single-tenant within-partition: 99/100 = 99%. Multi-tenant 5 clients: ~19%. Multi-tenant 20 clients: ~8%.
- **Architecture decision**: The optimization (static-first ordering) is free and should always be applied. But savings projections must use honest partition-adjusted hit rates, not single-tenant numbers.

### Key Test Patterns

**Deterministic mock judges**: Every bias test uses a callable with known behavior (e.g., `lambda q, a, b: "A"` for position-biased, `lambda q, a, b: "A" if len(a) > len(b) else "B"` for verbosity-biased). This proves the detection logic without LLM non-determinism.

**Boundary testing**: Drift severity tests exact thresholds (-0.02 is YELLOW, -0.019 is GREEN). This prevents off-by-one errors in classification logic.

**Scale testing**: 100-metric regression suite proves the system doesn't degrade at production scale. Cancellation masking test proves per-metric isolation.

### What ISN'T Tested (And Why)

- **Real LLM judge calls**: Phase 5 scope is the detection framework. Phase 12 validates against actual Claude/GPT judges.
- **Integration with Langfuse**: Drift alerts are generated as Python objects. Langfuse dashboard integration is Phase 12.
- **Production Redis cache metrics**: CacheMetrics is in-memory. Redis Stack integration is Phase 12.
- **Golden set staleness detection**: The 50-entry golden set needs quarterly review. Automatic staleness detection is Phase 8 scope.

## Interview Talking Points

1. **Judge independence at family level**: "We enforce family-level separation between judge and generator. GPT-5-mini judging GPT-5.2 is same-family self-preference — 10-15% score inflation in published benchmarks. The fix is a string prefix comparison that costs EUR 0.00 but changes the reliability of every quality metric. We'd revisit if providers prove no within-family self-preference via independent benchmarks."

2. **Position bias detection via pairwise swap**: "Every pairwise comparison runs twice with swapped positions. When rounds disagree, the result is discarded, not averaged. Averaging masks bias — the aggregate looks fine but individual results are unreliable. Bootstrap CI on the remaining results tells us if the reduced sample size is still trustworthy."

3. **Per-metric drift detection prevents aggregation masking**: "A 7% precision drop masked by a 2% relevancy improvement looks like -2.5% aggregate. Looks yellow. But precision dropping 7% is a RED alert. Per-metric detection catches this. We have a specific test that proves 5 regressions + 5 improvements still generate exactly 5 alerts."

4. **Correcting our own spec**: "The spec said 60% cache hit rate. We proved that RBAC partitioning fragments cache prefixes — each clearance-department-entity combination creates a separate partition. Multi-tenant reality: 15-25%. The optimization is still free, but the savings projections needed honest numbers. An architect who corrects their own spec earns more trust than one who confirms it."

5. **Fail-closed for unknown model families**: "Fine-tuned models and Azure deployment names don't match standard prefix patterns. They default to UNKNOWN, which fails the independence check. This is the safe default — blocking potentially biased judging is better than allowing it silently. Explicit registration (`register_model_family()`) overrides for known deployments."

6. **Spearman handles the edge case that breaks d²**: "We compute Spearman as Pearson-of-ranks instead of the classic d² formula. The d² approximation returns non-zero when one variable has zero variance (all scores identical). Pearson-of-ranks returns 0.0 — mathematically correct because correlation is undefined with zero variance."

7. **Bootstrap CI over Gaussian CI**: "Evaluation score distributions are often skewed — a model either gets a question right or completely wrong. Gaussian CI assumes normality. Bootstrap resampling works on any distribution. A score of 0.83 ± 0.09 reveals that the true value might be below the 0.80 quality gate, even though the point estimate passes."

8. **AlertHandler as extensible protocol**: "The drift detector generates alerts; handlers consume them. Adding Slack notifications is one method implementation. Adding PagerDuty is another. Detection logic never changes. This is the Observer pattern applied to infrastructure alerting."

## What I'd Explain Differently Next Time

**Start with the golden set, not the framework.** I built the detection logic first and the 50-entry golden set last. This meant writing all tests against synthetic mock judges first, then validating the golden set artifact separately. If I had the golden set on day one, I could have used real (human-scored) entries in tests from the start, which would have caught the Spearman tie-handling bug earlier.

**The "who watches the watchmen" framing should lead every conversation.** The technical details (pairwise swap, prefix matching, bootstrap resampling) only make sense once you understand WHY the evaluation pipeline needs its own quality assurance. Start with "your quality score might be inflated by 10-15%" and everything else follows.

**The RBAC-cache interaction is the best example of architect thinking in the phase.** It's a case where the spec was wrong and the implementation proved it with data. This pattern — "I found a structural reason the spec's claim doesn't hold, here's the honest number, and here's why the optimization is still worth doing" — is what separates architecture from implementation.

**The Spearman edge case taught me to always test degenerate inputs.** All-tied scores, zero-length inputs, single-element lists. The d² formula "worked" on normal inputs but broke on the one edge case that matters in production (when a model is uniformly bad and all human scores are 1s). Degenerate inputs reveal implementation bugs that happy-path testing misses.
